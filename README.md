## Why ASCII World?

The original Python ASCII-generator is wonderful, but it needs Python + OpenCV + NumPy + Pillow,
and it predates the age of AI agents. ASCII World is a ground-up rewrite designed for 2026 workflows:

| | ASCII World (Rust) | Original (Python) |
|---|---|---|
| Install | one static binary | Python + cv2 + numpy + Pillow |
| `img → txt`, 300 cols | **5 ms** | 253 ms |
| Machine-readable output | ✅ `--json` (lines + per-cell colors + alpha) | ❌ |
| Streams via stdin/stdout | ✅ | ❌ |
| MCP server for agents | ✅ built in | ❌ |
| Use as a library | ✅ crate API | ❌ |
| Input formats | PNG, JPEG, GIF, WebP, BMP, **SVG** | PNG, JPEG, … |
| Transparency | ✅ cutouts stay cutouts, `--bg transparent` | ❌ flattened to black |
| Images, GIFs, video | ✅ all three | ✅ all three |
| Runs in a browser | ✅ [WebAssembly](https://liohtml.github.io/ASCII-World/) | ❌ |

<sup>Benchmark: same source image, identical 300×82 output grid, average of 10 runs including process
startup, Intel Core Ultra 5 225U. Reproduce with [`docs/BENCHMARK.md`](docs/BENCHMARK.md).</sup>

## Quickstart

```bash
# install (Rust toolchain required — https://rustup.rs)
cargo install --git https://github.com/Liohtml/ASCII-World

# image → ASCII in your terminal
ascii-world txt photo.jpg --width 120

# true-color terminal art
ascii-world txt photo.jpg --ansi

# image → rendered ASCII PNG (like the gallery above)
ascii-world img photo.jpg -o art.png --color

# eight times the detail per cell
ascii-world txt photo.jpg --charset braille

# logos keep their transparency — cutout in, cutout out
ascii-world img logo.svg -o art.png --bg transparent --color

# animated GIF in, animated ASCII GIF out
ascii-world img clip.gif -o art.gif --color

# video (needs ffmpeg on PATH), with the source inset in the corner
ascii-world video clip.mp4 -o art.mp4 --color --overlay 0.2

# pipe-friendly: reads stdin, writes stdout
curl -s https://example.com/cat.png | ascii-world txt - --width 80
```

Run `ascii-world charsets` to see every built-in character set (`simple`, `complex`, `blocks`,
`braille`, 8 Latin/Cyrillic alphabets, `chinese`, `korean`, `japanese`) — or bring your own with
`--charset custom:"<chars dark→light>"`. `ascii-world txt --help` / `img --help` / `video --help`
list all flags: width, background, font size, inversion, alpha threshold, cropping, `--font`.

## 🤖 Use it from your agent (MCP)

ASCII World ships a built-in [Model Context Protocol](https://modelcontextprotocol.io) server —
no wrapper, no Node, no Python. Any MCP client gets an `image_to_ascii` tool:

```bash
# Claude Code
claude mcp add ascii-world -- ascii-world mcp
```

```json
// Cursor / Windsurf / generic MCP config
{
  "mcpServers": {
    "ascii-world": { "command": "ascii-world", "args": ["mcp"] }
  }
}
```

Your agent can now do things like *"convert build/hero.png to 120-column ASCII and put it in the
CLI's splash screen"* in one tool call.

**No MCP? JSON works everywhere.** `--json` emits the grid, the charset, and per-cell `#rrggbb`
colors — everything a downstream program (or LLM) needs to re-render the art in HTML, SVG, or a TUI:

```bash
ascii-world txt photo.jpg --width 60 --json | jq '{cols, rows, mode, first: .lines[0]}'
```

Transparent sources add an `alpha` array (mean 0–255 per cell) so a downstream renderer knows which
cells are holes rather than spaces.

There's also an [`AGENTS.md`](AGENTS.md) so coding agents can navigate this repo instantly.

## Use it as a library

```rust
use ascii_world::{charset, engine, input, render};

let charset = charset::resolve("complex")?;
let img = input::decode_still(&std::fs::read("photo.jpg")?, 800)?;
let grid = engine::convert(&img, &engine::Options {
    width: 100,
    charset: charset.clone(),
    ..Default::default()
})?;
println!("{}", render::to_text(&grid));
```

`input::decode_still` handles every input format including SVG (the second argument is the width to
rasterize vector input at). `engine::convert` takes an `RgbaImage`; anything you can build one of
works, so `image::open(..)?.to_rgba8()` is fine too.

Optional cargo features, all on by default: `cli` (the binary), `parallel` (rayon), `cjk` (the
bundled CJK font subset), `svg` (the rasterizer). Drop what you don't need — the wasm build uses
`--no-default-features --features wasm,cjk,svg`.

## How it works

1. The image is divided into a grid of cells (`--width` columns; cell height follows the font's real
   proportions when painting, or 2× width for terminals).
2. Each cell's average [Rec. 601 luma](https://en.wikipedia.org/wiki/Luma_(video)) picks a character
   from a **density-sorted ramp** — language charsets are sorted at runtime by rasterizing every
   glyph with the bundled fonts and measuring real pixel coverage. Cell colors are summed
   premultiplied by alpha, so transparent pixels never drag a cell toward black, and cells below
   `--alpha-threshold` come out blank whatever the ramp says.
3. `--charset braille` swaps step 2 for a 2×4 sub-cell grid per cell, placed with
   [Floyd–Steinberg](https://en.wikipedia.org/wiki/Floyd%E2%80%93Steinberg_dithering) error
   diffusion (which keeps both average tone and sub-cell structure) and packed into one
   U+2800–U+28FF codepoint. `--threshold N` switches to a plain hard cutoff instead.
4. Renderers turn the grid into plain text, ANSI true-color, JSON, or an RGBA image painted
   glyph-by-glyph (with value-normalized cell colors so colored output stays vivid), then trim the
   blank margins. Animation and video just run the whole pipeline per frame.

## Roadmap

Everything on the original roadmap has shipped:

- [x] `video2video` — feature parity with the original's video modes
- [x] Animated GIF output
- [x] Braille charset (`⣿`) for 2×4 sub-cell resolution
- [x] CJK charset presets (needs a bundled CJK-capable font)
- [x] Parallel cell sampling (rayon) for gigapixel inputs
- [x] Auto-crop of empty margins in PNG output (legacy parity)
- [x] WASM build — ASCII art in the browser
- [x] SVG input, rasterized internally
- [x] Alpha/transparency support — cutouts stay cutouts
